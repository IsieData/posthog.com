You can batch export data to S3-compatible object storage using two destination types:

- **AWS S3** - For buckets hosted on Amazon Web Services. Supports AWS-specific features like server-side encryption with KMS.
- **S3-compatible** - For S3-compatible providers like Cloudflare R2, MinIO, Google Cloud Storage, DigitalOcean Spaces, Hetzner, and others. Requires an endpoint URL.

## Choosing a destination

Use **AWS S3** when:
- Your bucket is hosted on Amazon Web Services
- You need AWS-specific encryption (AES256 or KMS)

Use **S3-compatible** when:
- Your bucket is hosted on a non-AWS provider (Cloudflare R2, MinIO, GCS, etc.)
- You need to specify a custom endpoint URL

## Models

This section describes the models that can be exported to S3.

> **Note:** New fields may be added to these models over time. Ensure any downstream processes can handle additional fields being added to the exported files.

### Events model

This is the default model for S3 batch exports. When exported in the Parquet file format, the schema is:

| Field             | Type        | Description                                                                                                  |
|-------------------|-------------|--------------------------------------------------------------------------------------------------------------|
| uuid              | `STRING`    | The unique ID of the event within PostHog                                                                    |
| event             | `STRING`    | The name of the event that was sent                                                                          |
| distinct_id       | `STRING`    | The `distinct_id` of the user who sent the event                                                             |
| person_id         | `STRING`    | The ID of the person the event is attributed to, resolved at ingestion time                                  |
| properties        | `STRING`    | A JSON object with all the properties sent along with the event, stored as a JSON-encoded string in Parquet  |
| person_properties | `STRING`    | A JSON object with the person's properties at ingestion time, stored as a JSON-encoded string in Parquet     |
| elements_chain    | `STRING`    | The chain of DOM elements for `$autocapture` events. Empty for other events                                  |
| timestamp         | `TIMESTAMP` | When the event occurred, as reported by the client. Microsecond precision, UTC                               |
| created_at        | `TIMESTAMP` | When PostHog ingested the event. Microsecond precision, UTC                                                  |
| _inserted_at      | `TIMESTAMP` | Internal field used by batch exports to track export progress. Included in files but safe to ignore          |

> **Note:** The types above describe the **Parquet** file format. Other file formats represent the same data differently. For example, in **JSONLines** exports the timestamp fields (`timestamp`, `created_at`, and `_inserted_at`) are ISO 8601 strings, and `properties` and `person_properties` are nested JSON objects rather than JSON-encoded strings.

### Persons model

The schema of the persons model when exported in the Parquet file format is:

| Field                      | Type        | Description                                                                                 |
|----------------------------|-------------|---------------------------------------------------------------------------------------------|
| team_id                    | `BIGINT`    | The ID of the project (team) the person belongs to                                          |
| distinct_id                | `STRING`    | A `distinct_id` associated with the person                                                  |
| person_id                  | `STRING`    | The ID of the person for this (`team_id`, `distinct_id`) pair                                |
| properties                 | `STRING`    | A JSON object with the latest person properties, stored as a JSON-encoded string in Parquet |
| person_distinct_id_version | `BIGINT`    | Internal version of the person-to-`distinct_id` mapping, used by batch exports during merges |
| person_version             | `BIGINT`    | Internal version of the person's properties, used by batch exports during merges            |
| created_at                 | `TIMESTAMP` | When the person was created. Microsecond precision, UTC                                      |
| _inserted_at               | `TIMESTAMP` | Internal field used by batch exports to track export progress. Included in files but safe to ignore |
| is_deleted                 | `BOOLEAN`   | Whether the person has been deleted                                                         |

Each export contains one row per `(team_id, distinct_id)` pair, mapped to their corresponding `person_id` and latest `properties`.

<PersonsModelNote />

> **Note:** As with the events model, these types describe the **Parquet** file format. In **JSONLines** exports, `created_at` and `_inserted_at` are ISO 8601 strings and `properties` is a nested JSON object rather than a JSON-encoded string.

You can view the schema for the sessions model in the configuration form when creating a batch export (there are a few too many fields to display here!).

## Creating a batch export

1. Go to [Data management > Destinations](https://app.posthog.com/data-management/destinations).
2. Click **+ New destination**.
3. Search for **AWS S3** or **S3-compatible** depending on your storage provider.
4. Click the **+ Create** button.
5. Fill in the [configuration details](#configuration).
6. Click **Create**.
7. The batch export schedules its first run on the start of the next period.

## Configuration

### AWS S3 configuration

| Field | Required | Description |
|-------|----------|-------------|
| **Bucket name** | Yes | The name of the S3 bucket where the data is exported |
| **Region** | Yes | The AWS region where the bucket is located |
| **Key prefix** | Yes | A key prefix for each S3 object created. Supports [template variables](#s3-key-prefix-template-variables) |
| **Format** | Yes | The file format for exported data (Parquet or JSON lines) |
| **Max file size (MiB)** | No | Split data into multiple files if it exceeds this size. If not set, data exports as a single file |
| **Compression** | No | Compression method for exported files (e.g., zstd, gzip) |
| **Encryption** | No | Server-side encryption method (`AES256` or `aws:kms`) |
| **AWS Access Key ID** | Yes | An AWS access key ID with access to the S3 bucket |
| **AWS Secret Access Key** | Yes | An AWS secret access key with access to the S3 bucket |
| **AWS KMS Key ID** | Conditional | Required when using `aws:kms` encryption |
| **Events to exclude** | No | A list of events to omit from the exported data |

### S3-compatible configuration

| Field | Required | Description |
|-------|----------|-------------|
| **Bucket name** | Yes | The name of the bucket where the data is exported |
| **Region** | Yes | The region for your storage provider |
| **Key prefix** | Yes | A key prefix for each object created. Supports [template variables](#s3-key-prefix-template-variables) |
| **Endpoint URL** | Yes | The endpoint URL for your S3-compatible storage provider |
| **Format** | Yes | The file format for exported data (Parquet or JSON lines) |
| **Max file size (MiB)** | No | Split data into multiple files if it exceeds this size. If not set, data exports as a single file |
| **Compression** | No | Compression method for exported files (e.g., zstd, gzip) |
| **Access Key ID** | Yes | An access key ID with access to the bucket |
| **Secret Access Key** | Yes | A secret access key with access to the bucket |
| **Virtual style addressing** | No | Enable if your provider requires virtual hosted-style addressing. Check your provider's documentation |
| **Events to exclude** | No | A list of events to omit from the exported data |

### S3 key prefix template variables

The key prefix can include template variables formatted at runtime. All template variables are defined between curly brackets (for example `{day}`). This allows you to partition files in your bucket by date or other attributes.

Template variables include:

- Date and time variables: `year`, `month`, `day`, `hour`, `minute`, `second`
- Name of the table exported: `table` (for example, 'events' or 'persons')
- Batch export data bounds: `data_interval_start`, `data_interval_end`

For example, setting `{year}-{month}-{day}_{table}/` as a key prefix produces files prefixed with keys like `2023-07-28_events/`.

### File formats

PostHog S3 batch exports support two file formats:

- [JSON lines](https://jsonlines.org/)
- [Apache Parquet](https://parquet.apache.org/) (latest version of the format specification)

We intend to add support for other common formats. You can follow the [roadmap](https://github.com/PostHog/posthog/issues/15997) to track progress.

### Compression

Each file format supports a variety of compression methods. We recommend using Parquet with zstd compression for the best combination of speed and file size.

> **Note on Parquet compression:** The compression type is included in the file extension, even for Parquet files. For example, files compressed with zstd have the extension `parquet.zst`. Since compression is embedded in the format itself, read the file directly as a Parquet file without uncompressing first.

### Manifest file

If you specify a max file size, several files may be exported. A `manifest.json` file (with the same prefix as the other files) is sent once all data files have been exported. This manifest file contains the key names of all the files exported.

## S3-compatible provider setup

This section provides configuration guidance for specific S3-compatible providers.

### Cloudflare R2

1. Select **S3-compatible** as your destination.
2. Set **Endpoint URL** to `https://<ACCOUNT_ID>.r2.cloudflarestorage.com` (replace with your Cloudflare account ID).
3. Select an R2 region from the **Region** dropdown, like "Automatic (AUTO)".

### MinIO

1. Select **S3-compatible** as your destination.
2. Set **Endpoint URL** to your MinIO instance's host and port (for example, `https://my-minio-storage:9000`).

### Google Cloud Storage (GCS)

Access to GCS requires a Service Account with HMAC credentials:

1. Follow the steps in the [BigQuery batch export documentation](/docs/cdp/batch-exports/bigquery) to create a Service Account.
2. Create a [HMAC key for your Service Account](https://cloud.google.com/storage/docs/authentication/managing-hmackeys#console).
3. Grant the Service Account the `Storage Object User` role or a custom role with at least these permissions:
   - `storage.multipartUploads.abort`
   - `storage.multipartUploads.create`
   - `storage.multipartUploads.list`
   - `storage.multipartUploads.listParts`
   - `storage.objects.create`
   - `storage.objects.delete`
4. Use the HMAC access key and secret key as **Access Key ID** and **Secret Access Key**.
5. Set **Endpoint URL** to `https://storage.googleapis.com`.
