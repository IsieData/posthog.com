---
title: Manage Workflows with MCP
---

The [PostHog MCP server](/docs/model-context-protocol) lets your AI coding agent create, manage, and test [Workflows](/docs/workflows) directly from your code editor. Build automation sequences, update configurations, and test executions – without switching to the PostHog app.

This works in any MCP client – Cursor, Codex, Claude Code, Windsurf, VS Code, and others.

## How it works

With MCP, your coding agent can:

- **Create workflows programmatically** – Build event-triggered automations, scheduled jobs, or webhook-driven flows from a prompt
- **Test workflow execution** – Run test invocations with mock data to verify your workflow behaves correctly before enabling
- **Update workflow configuration** – Modify triggers, actions, delays, and branching logic
- **Manage workflow lifecycle** – Archive workflows you no longer need, or disable active ones for maintenance

## Workflow tools

The MCP server provides these tools for working with Workflows:

| Tool | Description |
|------|-------------|
| `workflows-list` | List all workflows in the current project with their status, trigger type, and timestamps. |
| `workflows-get` | Get the full definition of a workflow, including trigger, actions, edges, and variables. |
| `workflows-create` | Create a new workflow as a draft with triggers, actions, delays, and branching logic. |
| `workflows-update` | Update a draft workflow's name, description, actions, or configuration. |
| `workflows-run` | Test-run a workflow with mock event data to verify execution before enabling. |

## Lifecycle tools

These tools manage workflow state transitions:

| Tool | Description |
|------|-------------|
| `workflows-enable` | Enable a draft workflow so it starts processing triggers. Prompts for confirmation. |
| `workflows-disable` | Disable an active workflow, returning it to draft state. Prompts for confirmation. |
| `workflows-archive` | Archive a workflow you no longer need. Cannot be undone via MCP. |

## MCP guardrails

The MCP server enforces a create-test-enable workflow to prevent accidental changes to live automations:

- **Workflows start as drafts** – You can't create an active workflow directly. Create it as a draft, test it with `workflows-run`, then enable it.
- **No editing active workflows** – You can't modify an enabled workflow's configuration. Disable it first, make changes, then re-enable.
- **Archive instead of delete** – Hard delete isn't available via MCP. Use `workflows-archive` to remove workflows from active use.

## Example prompts

Try these with your MCP-enabled agent:

- `List all my workflows and their status.`
- `Create a workflow that sends a Slack message when a user signs up.`
- `Show me the full configuration of my onboarding-email workflow.`
- `Test my welcome-sequence workflow with a mock signup event.`
- `Enable my welcome-sequence workflow now that testing passed.`
- `Disable my maintenance-alerts workflow so I can update it.`
- `Archive the old-drip-campaign workflow.`

## Install the MCP server

The recommended way to install is with the [AI wizard](/docs/ai-engineering/ai-wizard):

```bash
npx @posthog/wizard mcp add
```

The wizard supports Claude, Cursor, Windsurf, VS Code, and more. You can also [configure it manually](/docs/model-context-protocol#manual-install).

See the [MCP server docs](/docs/model-context-protocol) for full setup instructions.
