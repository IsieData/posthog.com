---
title: Manage Workflows with MCP
---

The [PostHog MCP server](/docs/model-context-protocol) lets your AI coding agent create, manage, and test [Workflows](/docs/workflows) directly from your code editor. Build automation sequences, update configurations, and test executions – without switching to the PostHog app.

This works in any MCP client – Cursor, Codex, Claude Code, Windsurf, VS Code, and others.

## How it works

With MCP, your coding agent can:

- **Create workflows programmatically** – Build event-triggered automations, batch broadcasts, or webhook-driven flows from a prompt
- **Test workflow execution** – Run test invocations with mock data to verify your workflow behaves correctly before enabling
- **Update workflow configuration** – Modify triggers, actions, delays, and branching logic
- **Run batch broadcasts** – Size an audience, confirm with the user, then trigger one-off or scheduled broadcasts
- **Debug workflow failures** – Identify failing workflows, inspect per-user invocations, and trace errors to specific steps
- **Manage workflow lifecycle** – Archive workflows you no longer need

## Workflow tools

The MCP server provides these tools for working with Workflows:

| Tool | Description |
|------|-------------|
| `workflows-list` | List all workflows in the current project with their status, trigger type, and timestamps. |
| `workflows-get` | Get the full definition of a workflow, including trigger, actions, edges, and variables. |
| `workflows-create` | Create a new workflow as a draft with triggers, actions, delays, and branching logic. |
| `workflows-update` | Update a draft workflow's name, description, actions, or configuration. |
| `workflows-test-run` | Test-run a workflow with mock event data to verify execution before enabling. |

## Batch workflow tools

Batch workflows broadcast messages to an audience of matching users. These tools manage batch execution with a required blast-radius confirmation step:

| Tool | Description |
|------|-------------|
| `workflows-blast-radius` | Count how many users match the workflow's audience filters. Always run this before `workflows-run-batch` or `workflows-schedule-create`. |
| `workflows-run-batch` | Trigger a one-off broadcast to all matching users. Requires the workflow to be enabled and a confirmed audience count. |
| `workflows-schedule-create` | Attach a recurring schedule (RRULE) to a batch workflow. Each scheduled run broadcasts to the current audience. |
| `workflows-list-batch-jobs` | List past batch runs for a workflow, including audience snapshots and timestamps. |
| `workflows-update-schedule` | Update a schedule's recurrence pattern, start time, timezone, or variable overrides. |

## Lifecycle tools

These tools manage workflow state transitions:

| Tool | Description |
|------|-------------|
| `workflows-enable` | Enable a draft workflow so it starts processing triggers. Prompts for confirmation. |
| `workflows-archive` | Archive a workflow you no longer need. Cannot be undone via MCP. |

## Debugging tools

Use these tools to identify and fix workflow failures:

| Tool | Description |
|------|-------------|
| `workflows-global-stats` | Get success and failure counts across all workflows in your project, sorted by most-failing first. Start here to identify which workflows need attention. |
| `workflows-stats` | Get success and failure metrics for a single workflow over time, including trend data. |
| `workflows-list-invocations` | List individual invocations (executions) for a workflow. Filter by status (`succeeded`, `failed`), user (`distinct_id`), or time range to find specific failures. |
| `workflows-get-invocation` | Get full details for a single invocation, including the triggering event payload, person data, and timing information. |
| `workflows-logs` | Retrieve execution logs for a workflow showing which steps ran and any error messages. |

The debugging tools form a natural triage path: start with `workflows-global-stats` to see which workflows are failing, drill into a specific workflow with `workflows-stats`, find affected users with `workflows-list-invocations`, inspect the triggering payload with `workflows-get-invocation`, and trace the failure to a specific step with `workflows-logs`.

## MCP guardrails

The MCP server enforces a create-test-enable workflow to prevent accidental changes to live automations:

- **Workflows start as drafts** – You can't create an active workflow directly. Create it as a draft, test it with `workflows-test-run`, then enable it.
- **Active workflows are read-only** – You can't modify an enabled workflow's configuration via MCP. Create a new draft workflow for changes instead.
- **Blast-radius confirmation for batch** – Before running a batch broadcast or creating a schedule, you must call `workflows-blast-radius` to size the audience, then confirm the count with the user. The MCP server rejects mismatched counts.
- **Archive instead of delete** – Hard delete isn't available via MCP. Use `workflows-archive` to remove workflows from active use.

## Example prompts

Try these with your MCP-enabled agent:

**Managing workflows:**

- `List all my workflows and their status.`
- `Create a workflow that sends a Slack message when a user signs up.`
- `Show me the full configuration of my onboarding-email workflow.`
- `Test my welcome-sequence workflow with a mock signup event.`
- `Enable my welcome-sequence workflow now that testing passed.`
- `Archive the old-drip-campaign workflow.`

**Batch broadcasts:**

- `How many users would receive the weekly-digest batch workflow?`
- `Run the announcement workflow now – I've confirmed the audience.`
- `Schedule my newsletter workflow to run every Monday at 9am.`

**Debugging failures:**

- `Which of my workflows are failing the most?`
- `Show me the failure rate for my onboarding-email workflow this week.`
- `List all failed invocations of my welcome-sequence workflow from the last 24 hours.`
- `What event triggered the failed invocation for user abc123?`
- `Show me the execution logs for my payment-confirmation workflow to find where it's failing.`

## Install the MCP server

The recommended way to install is with the [AI wizard](/docs/ai-engineering/ai-wizard):

```bash
npx @posthog/wizard mcp add
```

The wizard supports Claude, Cursor, Windsurf, VS Code, and more. You can also [configure it manually](/docs/model-context-protocol#manual-install).

See the [MCP server docs](/docs/model-context-protocol) for full setup instructions.
