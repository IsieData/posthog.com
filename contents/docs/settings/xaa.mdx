## XAA (Cross App Access)

XAA lets your tools, typically an AI agent or MCP client, automatically obtain PostHog access without the need 

With XAA, your IdP is used as a central authentication server. Your tools can then obtain access to PostHog by exchanging a cryptographic proof generated by your IdP for a short-lived PostHog API token. This API token can be used to access the PostHog API and MCP.

Under the hood, XAA uses the [Identity Assertion JWT Authorization Grant (ID-JAG)](https://xaa.dev) standard. Additionally, it is built on other open standards: the JWT Bearer grant ([RFC 7523](https://www.rfc-editor.org/rfc/rfc7523)), RFC 9068 access tokens, and RFC 8707 resource indicators.

## How XAA compares to SAML, SCIM, and OAuth

| | What it's for |
|---|---|
| **SAML** | Interactive, browser-based SSO login to the PostHog app. |
| **SCIM** | Automatically provisioning and de-provisioning user accounts and roles. |
| **OAuth** | A user interactively granting a third-party app access via PostHog's consent screen. |
| **XAA / ID-JAG** | An app or agent getting programmatic API access on a user's behalf, authorized by your IdP — no browser, no consent screen, no Personal API key. |

While you do **not** need SAML configured to use XAA, we recommend it. This ensures your IdP remains the source-of-truth for user authentication within your organization.


## Prerequisites

1. A  plan with XAA enabled. <FeatureAvailability availability={_frontmatter.availability.features.xaaAuthentication} />
2. A **verified authentication domain** (see [SSO docs](/docs/settings/sso#authentication-domains)). XAA configuration is per-domain and can only be saved once the domain is verified.
3. An OIDC identity provider that can issue ID-JAG tokens for cross-app access (for example, Okta or Microsoft Entra ID).

## Setup

### Posthog

1. Go to your [Organization settings → Authentication domains & SSO](https://us.posthog.com/settings/organization-authentication).
2. Click the **(⋯)** next to a verified domain and select **Configure XAA**.
3. Fill in the following:

| Field | Required? | Description |
|---|---|---|
| **IdP issuer URL** | ✅ Yes | Your IdP's issuer URL (e.g. `https://idp.example.com`). Must exactly match the `iss` claim on ID-JAG tokens for users on this domain. **XAA is not active until this is set.** |
| **JWKS URL** | ❌ Optional | Override key discovery. Leave empty and PostHog discovers keys via OIDC at `<issuer>/.well-known/openid-configuration`. |
| **Allowed client IDs** | ❌ Optional | Restrict which `client_id` values are accepted. Leave empty to allow any client. Useful if you only want to allow XAA in specific tools. |

4. Click **Save settings**.

To **deactivate** XAA for a domain, clear the IdP issuer URL and save.

> **Multiple domains:** XAA is matched by the user's email domain, so configure each verified domain with the IdP that serves that domain's users. A single domain can only point at one IdP issuer.

### IdP

Your IdP issues the ID-JAG token (header `typ: oauth-id-jag+jwt`). You must configure your IdP to set the correct claims in the ID-JAG token. PostHog validates these claims:

| Claim | Required? | Must be |
|---|---|---|
| `iss` | ✅ | Your IdP issuer URL — must match what you configured above. |
| `sub` | ✅ | The user's identifier. |
| `aud` | ✅ | Your PostHog URL (e.g. `https://us.posthog.com`). |
| `resource` | ✅ | Your PostHog URL (RFC 8707) — must equal `aud`. |
| `client_id` | ✅ | The app requesting access. Must be in **Allowed client IDs** if you set that list. |
| `scope` | ✅ | Space-separated PostHog [API scopes](/docs/api/overview#scopes) the IdP is granting (e.g. `insight:read feature_flag:read`). |
| `exp`, `iat` | ✅ | Standard expiry / issued-at. |
| `email`, `email_verified` | Optional | If `email_verified` is `false`, the token is rejected. |
| `jti` | Recommended | Used for replay prevention. |

> **`email` claim**: You must either provide the `email` claim, or ensure the `sub` claim is the user's email. If neither is an email, then the XAA token issuance will fail.

### Scopes

XAA uses the same scope vocabulary as PostHog's OAuth and Personal API keys. PostHog grants the **intersection** of the scopes the IdP authorized and any scopes the app requests at exchange time. If the intersection is empty a token is still issued, but API calls fail with `403 insufficient_scope` — so make sure you configure your IdP to grant any needed scopes.

## Security considerations

Please read these before enabling XAA.

1. **Only use XAA with an IdP that verifies user email addresses.** PostHog associates the token with a user by email; a spoofed email could impersonate a user. Tokens with `email_verified: false` are rejected.
2. **The user must be an active member of the organization** that owns the domain configuration. A valid token for a user who isn't in your org is rejected.
3. **Issued access tokens are short-lived** (2 hours by default) and bound to your organization, the granted scopes, and your PostHog URL. Replayed tokens are rejected via the `jti` claim.
4. **Enabling XAA does not disable Personal API keys or other auth.** XAA access tokens are accepted alongside existing credentials.

## Troubleshooting

XAA returns OAuth-style errors ([RFC 6749 §5.2](https://www.rfc-editor.org/rfc/rfc6749#section-5.2)).

| Error | Meaning / common cause |
|---|---|
| `invalid_request` | Missing/malformed `grant_type` or `assertion`, or wrong content type. |
| `unsupported_grant_type` | `grant_type` is not `urn:ietf:params:oauth:grant-type:jwt-bearer`. |
| `invalid_grant` (`"ID-JAG could not be verified"`) | The token failed verification — bad signature, expired, `iss`/`aud` mismatch, unverified email, user not an org member, or replayed token. The message is intentionally generic; check your IdP token's claims against the table above. |
| `invalid_client` | The token's `client_id` is not in your **Allowed client IDs** list. |
| `invalid_target` | The token's `resource` claim doesn't match your PostHog URL. |
| `access_denied` (403) | Your organization isn't entitled to cross-app access. Check your plan. |
| `403 insufficient_scope` (on API calls) | The access token's scopes don't cover the endpoint — grant the right scopes in your IdP. |
