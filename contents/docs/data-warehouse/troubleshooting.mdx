## Why is "Pull new schemas" not finding any tables?

When you click **Pull new schemas** on an external data source (Postgres, MySQL, Redshift, etc.) and see a warning that no tables were found, this typically means:

1. **Missing SELECT permissions** - The database user configured for the source doesn't have SELECT access to the target schema.
2. **Incorrect configuration** - The schema name, credentials, or connection details in your source settings are wrong.

To fix this, check that your database user has SELECT permissions on the target schema and its tables, and verify that your source credentials and configuration are correct.

## Why does "Pull new schemas" show "no schema changes"?

If you click **Pull new schemas** and see a success message saying there are no schema changes, this means the connection to your source is working correctly and all tables in the source schema are already being tracked.

This is expected behavior when your source configuration is up to date. New tables only appear if they've been added to the source database since your last refresh.

## Why am I seeing an error when pulling schemas?

When **Pull new schemas** fails, the error message shows the specific underlying cause instead of a generic failure message. Common errors include:

- **Authentication failures** - Invalid credentials or expired passwords.
- **Connection errors** - Network issues, firewall rules, or the database server being unreachable.
- **Permission denied** - The user lacks access to the specified schema or database.

Use the error details in the toast notification to diagnose and fix the issue in your database configuration.

## Why is my BigQuery sync failing with credential errors?

If your BigQuery sync fails with a message like "Your BigQuery service account credentials were rejected by Google," it means Google's token endpoint rejected the service account grant. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely.

This happens when:

- **The service account's private key was rotated or revoked** – Google returns an `Invalid JWT Signature` error.
- **The service account was deleted** – Google returns an `account not found` error.

To fix this:

1. Generate a new JSON key file for your service account in the [Google Cloud Console](https://console.cloud.google.com/) under **IAM & Admin > Service Accounts**.
2. In PostHog, go to [**Data pipelines > Sources**](https://app.posthog.com/data-management/sources) and select your BigQuery source.
3. Upload the new Google Cloud JSON key file.
4. Re-run the sync.

## Why is my BigQuery sync failing with a Storage Read API permission error?

If your BigQuery sync fails with a message like "Your service account is missing the bigquery.readsessions.create permission," it means the service account behind your BigQuery connection doesn't have the permission the Storage Read API needs to read your data. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since you'll need to grant the missing permission.

To fix this:

1. In the [Google Cloud Console](https://console.cloud.google.com/), grant the `bigquery.readsessions.create` permission to your service account. For example, assign the **BigQuery Read Session User** role on the project you're syncing.
2. Re-enable the source in PostHog.

## Why is my BigQuery sync failing with a billing error?

If your BigQuery sync fails with a message like "BigQuery billing is not enabled for your Google Cloud project," it means the Google Cloud project behind your connection has billing disabled (BigQuery sandbox mode), so BigQuery rejects every query job before it runs. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since you'll need to enable billing.

To fix this:

1. Enable billing for your project in the [Google Cloud billing console](https://console.cloud.google.com/billing).
2. Re-enable the source in PostHog.

## Why is my Snowflake sync failing with a "does not exist or not authorized" error?

If your Snowflake sync fails with a message like "A table or schema this source syncs no longer exists in Snowflake, or your role is no longer authorized to access it," it means a table or schema the source was syncing was dropped or renamed after it was discovered, or your role's access to it was revoked. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since you'll need to restore the object or its access.

To fix this:

1. In Snowflake, confirm the table or schema still exists and hasn't been dropped or renamed.
2. If it was renamed or removed, restore it or update your source to sync the current object.
3. If the object still exists, re-grant your Snowflake role access to it.
4. Re-enable the source in PostHog.

## Why is my Snowflake sync failing with an "invalid identifier" or invalid view error?

If your Snowflake sync fails with a message like "A Snowflake table or view you're syncing references a column that no longer exists" or "A Snowflake view in your source is invalid — the columns it declares no longer match the columns its query returns," it means a Snowflake object the source reads from won't compile. Either a table or view references a column that was dropped or renamed (an `invalid identifier` error), or a view's declared column list no longer matches the columns its query returns. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying can't compile a broken object until you repair it in Snowflake.

This is different from the "does not exist or not authorized" error above, where the object was dropped or its access revoked – here the object still exists, but its definition is broken.

To fix this:

1. In Snowflake, find the table or view this source reads from and check its definition.
2. Repair the broken object – update or remove the column the `invalid identifier` error names, or recreate the view so its declared columns match the columns its query returns.
3. Re-enable the source in PostHog.

## Why is my sync failing with "SSH tunnel credentials are not valid"?

If a source that connects through an SSH tunnel (Postgres, Redshift, MySQL, MSSQL, or ClickHouse) fails with a message like "Your SSH tunnel credentials are not valid," it means PostHog couldn't validate the SSH authentication details for your source – usually a private key that can't be parsed, or password authentication missing a username or password. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying would replay the same invalid credentials.

To fix this:

1. Open your source's settings and check the SSH tunnel authentication details – the private key and passphrase, or the username and password.
2. Correct any invalid or missing values.
3. Re-enable the source in PostHog.

## Why is my Meta Ads sync failing with "The Meta Ads integration for this source no longer exists"?

If your Meta Ads sync fails with a message like "The Meta Ads integration for this source no longer exists. Please reconnect the Meta Ads integration," it means the source still points at a Meta Ads integration that's been deleted or de-authorized, so PostHog can no longer find it. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying can't bring the integration back.

To fix this:

1. Reconnect the Meta Ads integration for the source.
2. Re-enable the source in PostHog.

## Why is my Postgres sync failing with "exceeded the compute time quota"?

If your Postgres sync fails with a message like "Your database provider has suspended the database because the account or project exceeded its compute-time quota," it means a serverless Postgres provider (like Neon) suspended your project's compute after the account used up its plan's compute-time quota, so PostHog can't connect until the database is available again. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since the compute stays suspended until you upgrade your plan or the quota resets.

To fix this:

1. Upgrade your database provider's plan, or wait for the compute-time quota to reset at the next billing period.
2. Re-enable the source in PostHog.

## Why is my Shopify sync failing with "402 Payment Required"?

If your Shopify sync fails with a message like "Shopify returned 402 Payment Required — your Shopify store appears to be frozen due to an unpaid bill," it means Shopify's Admin API returned a 402 because your store is frozen for billing reasons, usually a past-due or unpaid Shopify subscription. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since it can't succeed until the billing issue is resolved on Shopify's side.

To fix this:

1. Settle your outstanding balance in your [Shopify billing settings](https://admin.shopify.com/settings/billing) to unfreeze the store.
2. Re-enable the source in PostHog.

## Why is my Stripe sync failing with "does not have access to account"?

If your Stripe sync fails with a message like "Stripe rejected the request because your API key isn't authorized for the configured Stripe account," it means the API key can't reach the Stripe account the source is configured to use. This is usually because of an **Account id** (the `stripe_account` header) the key isn't authorized for, or a Stripe Connect application whose access was revoked. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying can't resolve a key/account mismatch or a revoked grant.

To fix this:

1. If your key belongs directly to the account, remove or correct the **Account id** in your source settings – it only applies to Stripe Connect platform accounts.
2. If you connected via OAuth, reconnect your Stripe account in case the application's access was revoked.
3. Re-enable the source in PostHog.

## Why is my Postgres sync failing with "the password that was provided for the role is wrong"?

If your Postgres sync fails with a message like "Your database proxy (for example AWS RDS Proxy) rejected the credentials," it means a database proxy in front of your Postgres database – such as AWS RDS Proxy – rejected the username and password the source connects with. Proxies report bad credentials in their own wording ("The password that was provided for the role ... is wrong") rather than PostgreSQL's standard authentication error. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying replays the same rejected credentials.

To fix this:

1. Check that the username and password configured for this source match what the proxy expects.
2. Re-enable the source in PostHog.

## Why is my Postgres sync failing with "no tenant identifier provided"?

If your Postgres sync of a Supabase database fails with a message like "Your Supabase connection pooler rejected the connection because it couldn't identify your project," it means Supabase's connection pooler (Supavisor) couldn't tell which project to route the connection to. On Supabase's shared pooler host, the connection username has to include your project ref (for example `postgres.<project-ref>`) – a plain username like `postgres` gives the pooler nothing to route on. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying replays the same username the pooler can't route.

To fix this:

1. Update the user for this source to the pooler username shown in your Supabase dashboard, in the format `postgres.<project-ref>`. You can find it under **Project Settings** → **Database** → **Connection pooling**.
2. Re-enable the source in PostHog.

## Why is my Postgres sync failing with "SASL authentication failed"?

If your Postgres sync fails with a message like "Your database rejected the credentials during authentication," it means the database server rejected the SASL/SCRAM credential exchange, usually because the username or password is wrong. This is most common with connection poolers like Supabase's transaction pooler (Supavisor), which report bad credentials as `FATAL: SASL authentication failed` rather than PostgreSQL's standard authentication error. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying replays the same rejected credentials.

To fix this:

1. Check that the username and password configured for this source are correct.
2. If you connect through a pooler (for example, Supabase's transaction pooler), make sure the username includes your project ref, in the format `postgres.<project-ref>`.
3. Re-enable the source in PostHog.

## Why is my GitHub sync failing with "No GitHub account is connected"?

If your GitHub sync fails with a message like "No GitHub account is connected. Please reconnect your GitHub account," it means the source is configured to authenticate with a GitHub account, but no connected integration was found – either no account was ever connected, or the linked integration was deleted. Related messages cover a missing personal access token or a GitHub integration that no longer exists. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely, since retrying can't restore a missing connection.

To fix this:

1. Reconnect your GitHub account for the source, or update the source configuration with a valid personal access token.
2. Re-enable the source in PostHog.

## Why is my Microsoft SQL Server sync failing with 'Invalid object name'?

If your Microsoft SQL Server sync fails with a message containing "Invalid object name," it means the database couldn't find an object (table or view) referenced in the query. This is a non-retryable error – PostHog stops the sync instead of retrying indefinitely.

This happens when:

- **A table or view was dropped or renamed** – The object you're syncing no longer exists.
- **A view references a missing object** – The view's underlying query references a table or another view that was deleted or renamed.
- **Cross-database permission issues** – The connection user can't access an object in another database that a view depends on.

To fix this:

1. Check that all tables and views you're syncing still exist in your database.
2. If you're syncing views, verify that all objects referenced in the view definitions are accessible.
3. Ensure your connection user has SELECT permissions on all required objects, including cross-database dependencies.
4. In PostHog, go to [**Data pipelines > Sources**](https://app.posthog.com/data-management/sources) and re-sync after fixing the issue.

## How do I speed up my SQL queries?

1. Reduce the date range you're querying.
2. Add more specific filters with `WHERE` clauses.
3. Use [materialized views](/docs/data-warehouse/views/materialize).

We have other suggestions in our [product analytics docs](/docs/product-analytics/troubleshooting#insights-and-queries-faq).

## How do I run partial SQL queries?

If you want to run a part of your query in the SQL editor, highlight the part you want to run and hit **CMD + Enter** (Mac) or **CTRL + Enter** (Windows).

This is useful if you have a complex query and you want to test parts of it without running the whole query.

## Why is my CSV data not parsing correctly?

If your CSV data appears garbled, has missing columns, or shows unexpected characters, it may be due to incorrect quote handling settings.

When linking a CSV file, PostHog provides a **CSV quote handling** option with two modes:

- **RFC 4180 double quotes** - Use this when your CSV uses standard double-quote escaping (e.g., `"Hello ""World"""` becomes `Hello "World"`)
- **Literal quotes** - Use this when quotes in your data should be treated as regular characters (default)

To fix parsing issues, try recreating the table with the other quote handling option selected.

## Why am I getting "Some files in the bucket are archived"?

This error occurs when files in your S3 bucket are stored in S3 Glacier or S3 Intelligent-Tiering archive storage classes. These storage classes are designed for long-term archival and cannot be accessed directly.

To resolve this:

1. Restore the files to Standard storage class in AWS S3.
2. Or narrow your URL pattern to exclude the archived files.
